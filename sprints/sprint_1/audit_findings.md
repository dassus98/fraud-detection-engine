### Gap-fill applied

**One docstring clarification on `TransactionCleaner.clean()`** — and the reason it matters is concrete: I hit it myself during the 1.2.d build_interim work. The original docstring said *"input must satisfy `MergedSchema`"* but a careful reader still wouldn't realise that calling `loader.load_merged(optimize=True)` produces output that does **not** satisfy `MergedSchema` (the loader validates *before* its `_optimize` step, then downcasts to int32/float32/category and returns; subsequent `cleaner.clean()` then re-validates against `InterimTransactionSchema` which inherits the strict dtypes via `MergedSchema.add_columns`, and pandera raises `SchemaErrors` mid-clean).

Concretely: when I built `scripts/build_interim.py` in the 1.2.d prompt, the first run failed with seven pandera dtype mismatches because I had naïvely passed `optimize=True`. I worked around it by using `optimize=False` and documented the finding in the 1.2.d completion report. **A future caller would hit the exact same wall.** The clarification I added to the docstring spells out the constraint and the exact failure mode, so the next reader sees it before running.

The change is purely additive (no behavioural delta, no API change). The 11 unit tests still pass; mypy clean; ruff format reflowed the new lines but no other content moved.

### Latent design concern (not gap-filled this audit)

There is a deeper design issue worth flagging for a Sprint 2 housekeeping pass: `InterimTransactionSchema` inherits `MergedSchema`'s **strict** dtype constraints, which couples the cleaner's exit contract to the loader's pre-optimise stage. A cleaner architecture would either:

- **(A)** Define `InterimTransactionSchema` independently with looser dtype family checks (e.g., any int dtype rather than int64 specifically), so it accepts both un-optimised and optimised frames;
- **(B)** Have the cleaner re-apply the loader's `_optimize` after validation (or import the helper), so the interim layer is always optimised on disk;
- **(C)** Document the cleaner as un-optimised-input-only (current state, now made explicit by this audit's docstring change).

**(C) is what we have today and is sufficient.** (A) is the right architectural fix but touches `MergedSchema` and ripples through every test fixture; out of scope for "audit and gap-fill". (B) is feasible but adds private-helper coupling.
