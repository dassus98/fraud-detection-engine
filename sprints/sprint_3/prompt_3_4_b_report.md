## Decisions worth flagging

1. **`pyg-lib` CPU wheel via `[tool.uv.sources]` URL pin.** PyG's neighbor-sampling backend requires either `pyg-lib` or `torch-sparse`; the in-tree pure-Python fallback is gone in current `torch_geometric==2.6.1`. The CPU wheel for `pyg-lib==0.4.0+pt25cpu` is hosted on `data.pyg.org` (torch-version-and-platform-specific), not PyPI. We use uv's `[tool.uv.sources]` to pin the wheel URL deterministically. This makes `uv sync --all-extras` reproducible without out-of-band install steps. Trade-off: the source pin is Linux-x86_64 + CPython 3.11 specific (mirrors the rest of the project's environment); a contributor on macOS or arm64 would need a different wheel. Documented inline.

2. **Homogeneous PyG `Data` (not `HeteroData`).** Both transaction and entity nodes live in the same `x` tensor (entity rows zero-padded). Sidesteps the `to_hetero(...)` rewrite, keeps `SAGEConv` simplest, and makes the bipartite-invariant test trivial (XOR of `src < n_txn` and `dst < n_txn`). Cost: ~41 MB of zero-padding at IEEE-CIS scale; acceptable.

3. **Transductive node-classification with masks.** Graph built ONCE from train+val+test entities; train mask labels train txns; val/test masks slice the rest. Loss is masked. Labels for val/test never enter training — only the topology + features are visible (textbook PyG node-classification pattern). CLAUDE.md §3 calls Model C "batch-only", so transductive is the right contract. **Inductive scoring** (predicting on TransactionIDs not in the persisted graph) **raises `KeyError`**; integration test enforces this. Sprint 5+ may add inductive support.

4. **Cached node logits at fit-time** (rather than per-call forward pass). After training, the model runs ONE full-graph forward pass and stores per-node logits on `self.cached_node_logits_`. `predict_proba(df)` becomes a TransactionID-index lookup + sigmoid. Latency p95 = 0.072 ms versus the multi-second per-call cost a naïve subgraph-extraction-per-request would incur. Sprint 5 (real-time serving) would replace this with a subgraph-extraction-per-request implementation; Sprint 3 (batch-only per CLAUDE.md §3) doesn't need it yet.

5. **`FocalLoss` reused from `neural_model.py`.** No duplication. `from fraud_engine.models.neural_model import FocalLoss` at the top of `gnn_model.py`. Sprint 4 calibration toolkit similarly reusable.

6. **Module-scoped `smoke_result` fixture** (lessons from 3.4.a). Function-scoped would re-train per fixture-using test. Module scope means the suite trains the GNN exactly once. **8 tests in 8.53 s.**

7. **`P_emaildomain` excluded from the entity columns** (despite being in `TransactionEntityGraph`). Email-domain has only 59 unique values × 414K txns ≈ 7K txns per domain; making it a graph node would create high-degree hubs that dominate the message-pass and dilute the structural signal that card1/addr1/DeviceInfo carry. Documented inline.

8. **`scale_pos_weight` not used.** Focal loss is the imbalance handler. Mixing both would double-weight the minority class. Same trade-off as 3.4.a.

9. **Default 5K-smoke (--quick) and 50K (default) subsamples** — same approach as 3.4.a. The 4-hour cap and the spec's optional-deferral language are honored by sizing the production run to fit any reasonable WSL/CPU environment. `--full` exposed for stable environments.

## Surprising findings

1. **Latency is essentially zero.** p50 = 0.037 ms = 37 μs. The cached-logits design makes `predict_proba` purely a Pandas `.iloc` + numpy fancy index. This is what production batch scoring SHOULD look like — Sprint 5's batch-feature pipeline can score ~1M rows in under 1 second.

2. **Val AUC climbs steeply on the smoke** (0.4546 epoch 1 → 0.7778 epoch 5). The 3-layer SAGEConv learns to leverage the bipartite topology fast on 5K rows. With more epochs and the 50K production run, we expect the trajectory to extend further. The smoke's no-early-stop result indicates we should run more epochs to hit the natural convergence point for full training.

3. **The bipartite-invariant test is an XOR check** that completes in microseconds (single tensor op); it's a strong correctness check and doubles as a sanity-of-graph-construction test. Future graph-derived models can reuse the same pattern.

4. **`pyg-lib` install path is non-trivial** but uv's `[tool.uv.sources]` URL pin makes it reproducible. This is the cleanest reproducibility-preserving solution short of vendoring the wheel into the repo.

## Out of scope (Sprint 4+ territory)

- **Inductive scoring** of new TransactionIDs (Sprint 5 serving deliverable). The contract is documented in `predict_proba`'s `KeyError` path.
- **Calibration of FraudGNN outputs.** 3.3.c toolkit reusable in Sprint 4.
- **Ensemble with Models A + B.** Sprint 4 deliverable. The error decorrelation hinted at by FraudGNN's competitive smoke result is the precondition.
- **GNN explainability** (e.g., GNNExplainer). Out of scope.
- **HeteroData / `to_hetero(...)` restructuring.** Sprint-3 minimum-viable; the homogeneous-Data path is the simplest correct option.
- **GPU code path / distributed training.** A `device=` knob exists; only `cpu` is exercised.
- **Edge-feature engineering** (e.g., transaction recency on each edge). Out of scope.
- **Full-data training run (414K rows)** on a stable shell environment. The `--full` flag exposes the path; the 5K-smoke (--quick) and the integration test verify the architecture; the 50K canonical (default) is the production run. Re-running the `--full` path on a stable environment is mechanical.
