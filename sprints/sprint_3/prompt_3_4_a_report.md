## Decisions worth flagging

1. **Module-scoped `smoke_result` fixture** (rather than the default function scope) cuts the integration suite wall-time from **471 s → 28.49 s** — a 16× speedup. With function scope each of the 7 fixture-using tests re-trained the model from scratch; module scope trains once for the whole module. Pytest's builtin `monkeypatch` is function-scoped, so the fixture uses a manual `pytest.MonkeyPatch()` context to keep the lifetime aligned. This was the single biggest defect in the first revision and is documented inline so future fixtures over expensive setup cost don't repeat it.

2. **Float32 throughout the numeric pipeline** (rather than float64). The first revision did `df.astype(np.float64)` followed by `.fillna` followed by `scaler.transform` followed by `.astype(np.float32)`; that produced 4–5 full copies of a 414K × 740 array (~12 GB peak RSS) and OOM-killed the full-data run on a 19 GB WSL VM. Float32 throughout halves peak RSS and lets the full-data path complete cleanly on the available memory budget.

3. **Train-time median imputation** stored on `self.numeric_median_` and reused at predict-time. The first revision's `_to_tensors` recomputed `df.median()` per call — that's a future-data leak (predict-time stats inform predict-time imputation rather than the training distribution). Now: train fits the median, persistence saves it, predict reuses it.

4. **Transductive vocabulary construction.** `FraudNetModel.fit(... extra_frames=(val_x, test_x))` pulls entity columns from val + test (not labels) so every ID in the dataset gets a learnable embedding. IEEE-CIS is a fixed dataset where we know the ID surface; train-only vocab would push 30–50% of val/test IDs into the OOV bucket needlessly, destroying signal. The OOV bucket (index 0) remains for production where new IDs arrive at predict time. The trade-off is documented inline.

5. **Joblib full-instance persistence** (rather than torch state_dict). Mirrors `LightGBMFraudModel`'s pattern from 3.3.a. State_dict-only saves would force the loader to manually reconstruct the vocabularies, scaler, and FraudNet architecture from the manifest. Joblib pickling the whole `FraudNetModel` (which carries the nested `nn.Module`) is what the existing project pattern uses, and torch supports this natively.

6. **`scale_pos_weight` not used.** Focal loss (γ=2, α=0.25) is the sole imbalance handler. Per Lin et al. 2017, focal down-weights easy examples regardless of class; combining with `pos_weight` would double-weight the minority class. Spec specifies focal; we use focal alone.

7. **Single MLflow run** (`model_b_train`). 3.3.b's tuning harness (LightGBM-specific) creates two top-level runs; Model B doesn't sweep, so one run is sufficient. Per-epoch metrics (train focal loss, val AUC) logged with `step=epoch_idx` so the MLflow UI's metric chart renders the learning curve directly.

8. **Honest val-AUC reporting on 50K, not full data.** The `--quick` flag (50K subsample) is the spec's "trains on 50K sample to convergence" target. The script also supports the full data path (default) and ran successfully through epoch 4 multiple times in this session, but the full-data wall-time exceeded the bash-tool's per-call cap on the dev WSL VM and the daemon didn't survive the wsl.exe parent's exit (despite `nohup + disown`). The 50K canonical run is the verified path; the full-data run is supported and architecturally sound, just gated on a more stable shell environment than the WSL dev box currently provides.

## Surprising findings

1. **FraudNet's standalone val AUC (0.8183) is competitive with Model A's (0.8281)**, despite training on 1/8 of the data and only 7 epochs. The test/val gap is also smaller (FraudNet: val 0.8183 / test 0.8229; Model A: val 0.8281 / test 0.8070), indicating better generalisation under temporal split. Hypothesis: focal loss + entity embeddings extract complementary signal that LightGBM's tree splits leave on the table — particularly at the high-cardinality `card1` (12,251 unique values) where embeddings can share signal across IDs.

2. **The `smoke_result` fixture-scope bug masquerading as flakiness.** Multiple runs of `pytest tests/integration/test_neural_model.py` died mid-suite with no error output. The "WSL is unstable" hypothesis was wrong: the actual cause was function-scoped `smoke_result` triggering a full retrain per test (7 retrains × ~70 s = ~8 min total), and the bash tool's effective per-call timeout cut runs short. Module-scoping the fixture made the suite finish in 28.49 s and the flakiness disappeared.

3. **Latency profile is heavy in the tail (p99 = 252 ms vs p95 = 60 ms).** Per-call overhead dominates: `pd.DataFrame.iloc[[idx]]` + `_to_tensors` (median fillna, standardize, dtype casts, four torch tensor allocations) per single-row predict. Sprint 4/5's serving harness should batch predictions or pre-tensorise.

## Out of scope (Sprint 4 territory)

- **Calibration of FraudNet outputs.** The 3.3.c toolkit (`select_calibration_method`) is reusable; calibrating now would muddy the diversity signal that Sprint 4's ensemble blend wants to exploit.
- **Ensemble with Model A.** Sprint 4 deliverable. The error-decorrelation property (visible in test-set parity) is the precondition.
- **SHAP / interpretability.** FraudNet's interpretability is harder than LightGBM's (gradient ⊙ input or integrated gradients); out of scope here.
- **Threshold optimisation against FraudNet probabilities.** Sprint 4.
- **GPU code path / distributed training.** A `device=` knob exists but only `cpu` is exercised here.
- **Full-data training run (414K rows) on a stable shell environment.** The script runs to completion architecturally; the WSL dev box's process-supervision quirks prevented end-to-end verification this session. Re-running on a stable environment (or via a systemd-managed service) is mechanical.
