## Decisions worth flagging

1. **Two top-level MLflow runs (tuning + final-fit) instead of one nested hierarchy.** `run_tuning` from 3.3.b already opens its own `mlflow.start_run` for the tuning study. Wrapping that under a parent would require an API change to 3.3.b. Two top-level runs in the same experiment make the UI's "Compare runs" view immediately productive without breaking the 3.3.b contract.

2. **`select_calibration_method` chose isotonic.** The LightGBM output's miscalibration shape (overconfidence at extremes; under-confidence in the middle, exacerbated by `scale_pos_weight=27.4` on a 3.5 % base rate) is non-sigmoid. Platt's 2-parameter logistic can't fit it; isotonic's piecewise-constant non-decreasing map can. Realised: val log_loss 0.291 → 0.109 (62 % better); val ECE 0.193 → 0.000.

3. **Honest val-AUC gap reporting.** Per the approved option-1 plan, the realised AUC of 0.8281 is reported as-is. The Sprint 1 baseline (Tier-1 only) hit 0.9247; the regression to 0.8281 with the full Tier-5 stack at tuned hparams indicates a feature-engineering issue, not a tuning issue. Sprint 4 follow-on candidates: feature pruning (the 743-column space dilutes the top signals), alternate val-split strategies, model diversity (FraudNet / FraudGNN per CLAUDE.md §3).

4. **Trial history is empty in the report.** The 3.3.b `run_tuning` harness encapsulates the Optuna study and writes only `best_params` + `best_value` + `n_trials` to YAML. The training script doesn't get the per-trial values back. The MLflow tuning run carries every trial as a child run; the report cross-references that. Improving the harness to return the study (or the trials_dataframe) is a small follow-on.

5. **Sample-size override flag (`--quick`)** for the integration test smoke. 5 trials × 5k stratified subsample × small boosting cap (~30 rounds) = ~50 s wall-time. Same code path as production; no separate code branches.

6. **`models/sprint3/` is the canonical artifact directory.** Sprint 4's cost-curve evaluator + Sprint 5's serving stack will load from this path. `LightGBMFraudModel.save` writes `lightgbm_model.joblib` + manifest; the calibrator joblib lives next to it.

## Surprising findings

1. **Calibration's log-loss improvement is dramatic.** The 0.291 → 0.109 collapse (62 % better) is much larger than the spec's "calibration doesn't hurt" gate anticipated. Cause: the LightGBM model fitted with `scale_pos_weight=27.4` on a 3.5 % base rate produces probabilities that are systematically overconfident at the extremes; isotonic regression corrects this directly. Sprint 4's threshold sweep on calibrated probabilities will operate on a much faithful cost surface than the uncalibrated path would have allowed.

2. **`pagerank_score` (Tier-5) ranks 5th in feature importance.** Higher than expected — graph features are typically tail-importance signals on top of strong velocity/EWM stacks, but pagerank's structural-hubness signal complements the per-card EWM features in a way the model exploits. Confirms 3.2.b/c's design hypothesis at the integrated level.

3. **The val-AUC gap is consistent with the Tier-2-5 regression theme.** Sprint 1 baseline (Tier-1 only) was 0.9247; every subsequent tier added at default hparams degraded AUC. Tuning recovered ~+0.06 (0.7689 → 0.8281) but couldn't close the gap to Sprint 1 baseline. **Hypothesis:** the Tier-5 graph layer's structural features (CC size, clustering, pagerank) contribute usable signal but the bulk of the 743-column space is noise that confuses LightGBM's split-finding. Feature pruning by gain importance + retraining is the obvious Sprint 4 experiment.

4. **Test/val gap is modest (val 0.828 vs test 0.807, ~−0.02).** The model generalises reasonably from val to test. The temporal gap doesn't introduce major drift — the Tier-4 EWM + Tier-5 graph features hold up under the time shift.

## Out of scope (Sprint 4 territory)

- **Feature pruning + retrain.** Drop low-importance features (the 743-column space is heavy on Vesta V-features that may not survive across temporal splits), retrain. Plausible AUC recovery to 0.93+.
- **Threshold optimisation** against the calibrated probabilities (Sprint 4 deliverable).
- **Cost-curve evaluation** + economic-cost minimisation.
- **Stratified metrics** by amount bucket / ProductCD / time bucket.
- **Trial-history enrichment in the report** — minor follow-on; needs a small `run_tuning` API change to return the study object.
